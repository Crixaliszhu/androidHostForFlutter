# 最佳实践评估与演进建议

## 1. 文档目标

本文档基于当前仓库实际代码，对鱼泡直聘混合项目现有技术方案做最佳实践评估，并给出可执行的演进建议。

这里讨论的不是“理论上最完美的绿地架构”，而是“大型存量混合项目在现实约束下的最优演进方向”。

## 2. 总体结论

当前方案不是理论最优，但已经接近大型存量业务的务实最优：

- Android 宿主平台化方向是对的
- RN 使用契约层和新架构接口是对的
- Flutter 使用 Pigeon 和 Add-to-App 分层是对的
- 原生 data/domain/feature 分层方向是对的

主要问题不在方向错误，而在于：

- 新旧架构并存时间较长
- 路由和存储口径过多
- 双前端技术栈长期并行导致复杂度较高
- 平台层与业务层边界仍有部分混杂

## 3. 分项评估

## 3.1 Android 宿主与原生架构

### 当前评价

`recruitment_android` 的整体结构是合理的，尤其是：

- 宿主职责集中
- 模块化程度高
- Hilt 已经落地
- 新业务有 data/domain/feature 分层
- RN/Flutter 已经被独立模块承载

### 与最佳实践相比的差距

- 存储方案偏多
- 路由栈偏多
- DataBinding 等老方案仍占据一定比重
- 少量历史工具类调用方式不够收敛
- `rn_base` 平台层承担了较多业务耦合依赖

### 建议

1. 新功能统一进入 `feature + domain + data + api/model` 路线。
2. 平台模块继续去业务化。
3. 路由门面统一，底层实现隐藏。
4. SharedPreferences 逐步迁移。
5. DataBinding 新增场景停止扩散。

## 3.2 Flutter 方案

### 当前评价

Flutter 侧的工程化程度高于很多常见 Add-to-App 项目：

- Pigeon 使用正确
- 路由体系有中心化意识
- 数据层已经分层
- MMKV 共享方案落地
- 工程规则、生成器、lint 都在建设

### 与最佳实践相比的差距

- 自定义基础设施较重
- 平台桥封装数量多，后续维护成本可能上升
- 路由兼容映射和跨容器判断逻辑较复杂
- 状态管理没有形成一个足够清晰、易传递的统一心智

### 建议

1. 保留 Pigeon 作为主桥接方式。
2. 将 bridge 能力按业务域归并，避免无限扩张。
3. 路由协议继续收敛，减少兼容散点。
4. 明确 Flutter 侧页面状态组织规范，降低新人接入成本。
5. 若后续 Flutter 占比进一步提升，再评估更统一的状态与依赖方案。

## 3.3 RN 方案

### 当前评价

RN 侧在存量项目里属于比较积极地向现代化靠拢：

- RN 0.72
- Hermes
- New Architecture 已启用
- TurboModule 和 codegen 方案已接入
- 契约层独立出来

### 与最佳实践相比的差距

- Redux Toolkit 与 MobX 并存
- TypeScript 严格度偏低
- 页面组织中仍有较重的 class VM / decorator 历史风格
- CodePush 长期依赖需要持续治理

### 建议

1. 明确状态分工：
   - Redux Toolkit 负责全局与可序列化状态
   - MobX 仅限复杂交互页
2. 新模块逐步提高 TS 约束。
3. 优先用 hooks、typed selector、typed service 组织新页面。
4. 保持 `ypmoduledecorate` 作为唯一对外桥接契约层。
5. 持续治理 CodePush 与版本兼容风险。

## 3.4 混合项目总体方案

### 当前评价

当前混合方案的优点是：

- 兼容历史业务
- 允许多技术栈逐步演进
- 宿主层仍保持控制权
- 跨端能力已经开始平台化

当前混合方案的主要问题是：

- 双前端主栈长期并行，复杂度高
- 同一业务域存在重复实现风险
- 路由、结果回传、AB、兼容层复杂
- 平台层容易被业务侵入

### 更接近最佳实践的目标形态

更优的目标形态应该是：

- 宿主只负责平台控制
- 原生只承载强系统耦合与核心宿主链路
- RN 与 Flutter 至少有一方逐步收敛为主力前端技术栈
- 路由、埋点、缓存、账号协议统一
- 三端边界模型统一

## 4. 不建议做的事情

基于当前阶段，不建议做以下动作：

1. 一次性大规模重构为单技术栈。
2. 为追求“新”而在存量项目里强行全面切 Compose、Bloc、Riverpod 或完全重写 RN 状态体系。
3. 在没有统一协议前继续增加新的散乱桥接方式。
4. 在宿主层继续堆叠具体业务逻辑。

这些动作短期收益低，风险高，且容易放大现有复杂度。

## 5. 建议的演进路线

## 5.1 短期建议

目标：控复杂度，不新增新债。

建议：

1. 新需求先明确归属端，不做无必要跨端重复开发。
2. 新增桥接必须走契约层：
   - RN 走 `ypmoduledecorate`
   - Flutter 走 Pigeon
3. 新增原生模块优先遵循 `data/domain/feature`。
4. 新增存储优先使用既定规范，不再引入新的本地存储口径。

## 5.2 中期建议

目标：做平台收敛。

建议：

1. 收敛统一路由门面。
2. 收敛统一账号态与缓存态协议。
3. 将 `rn_base`、`flutter_biz` 进一步平台化。
4. 梳理同一业务域在三端的重复实现清单。

## 5.3 长期建议

目标：做技术主栈治理。

建议：

1. 评估未来主力前端栈的方向。
2. 对重复业务域逐步迁移和收敛。
3. 形成真正统一的跨端平台层：
   - 路由协议
   - 事件协议
   - 账号协议
   - 缓存协议
   - 埋点协议

## 6. 推荐的判断标准

后续评估一个技术方案是否值得继续推进，建议看四个标准：

1. 是否降低了业务新增复杂度。
2. 是否降低了跨端协作复杂度。
3. 是否减少了平台层与业务层耦合。
4. 是否减少了三端重复实现。

如果一个方案不能同时改善这几个方向，就不算对当前项目真正有价值的最佳实践。

## 7. 最终判断

当前鱼泡直聘混合项目的技术方案，结论不是“落后”，也不是“最佳”，而是：

`方向基本正确，但需要持续做边界收敛、协议统一、平台去业务化和技术主栈治理。`

对这个项目而言，真正的最佳实践不是推翻，而是收敛。
